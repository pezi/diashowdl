§1Encoding Format

DDL files are valid JSON documents parsed by the Flutter runtime via dart:convert. For distribution and transfer, DDL shows can be packaged as DDLZ archives (see §12).

PropertyJSONDDLZ
File extension.ddl.json.ddlz
Primary useRuntime / API delivery / authoringSelf-contained distribution / transfer

§2Document Structure

Every DDL document is a single JSON object with four top-level keys:

{
  "ddl":      "0.1",              // REQUIRED. DDL version string.
  "meta":     { ... },        // OPTIONAL. Show-level metadata. See §2.2.
  "defaults": { ... },        // OPTIONAL. Global fallbacks for all slides.
  "show":     [ ... ]          // REQUIRED. Ordered array of slide objects.
}

2.1 Inheritance Model

Each slide property is resolved by looking first at the slide itself, then falling back to defaults, then to the DDL built-in defaults shown in §4. This means a show with identical transitions can set them once in defaults and override only the exceptions per-slide.

2.2 Metadata

The optional meta object provides descriptive information about the show, analogous to HTML <meta> tags. Metadata is not interpreted by the DDL player at runtime — it is informational and intended for cataloguing, search indexing, or external tooling.

PropertyTypeDescription
namestringHuman-readable name / title of the show.
descriptionstringShort description of the show's content or purpose.
keywordsstringComma-separated keyword list for search / categorisation.
authorstringName or identifier of the show's author.
any keystringArbitrary additional metadata entries. The parser preserves all unknown keys.
 
"meta": {
  "name":        "European Sunsets",
  "description": "A sunset timelapse across European capitals",
  "keywords":    "sunset, timelapse, europe, capitals",
  "author":      "Jane Doe",
  "created":     "2026-03-01",
  "license":     "CC BY 4.0"
}

3.3 Lottie Animations

When "type" is set to "lottie", the src must point to a Lottie JSON animation file (either a local asset or a network URL). The animation loops continuously for the slide's duration. Ken Burns effects are not applied to Lottie slides. The fit property controls how the animation fills the viewport (defaults to "contain" for Lottie to preserve animation proportions).

{
  "type": "lottie",
  "src":  "asset://animations/intro.json",
  "duration": 5000,
  "fit": "contain"
}
Implementation Note The Flutter runtime uses the lottie package (v3.3.x). Asset sources are rendered via Lottie.asset(); network sources via Lottie.network(); file sources (from DDLZ extraction, §12) via Lottie.file(). The animation repeats in a loop with repeat: true.

3.4 Web & PDF Slides — Display-Only by Design

Diashow is a presenter, not a remote-maintenance or browsing tool. When "type" is set to "web" or "pdf", the embedded content is rendered for display only — all user input is intentionally suppressed. Scrolling, link clicks, form input, page-turn gestures, pinch-zoom, and text selection are all blocked at the slide level.

This is a deliberate runtime guarantee, not a missing feature. It exists so that in remote presenter sync mode (§9) the server (display) and client (presenter) always render the same frame: neither side can diverge by interacting with the embedded web page or PDF. Slide advancement, timing, transitions, and the laser pointer remain driven entirely by the DDL show and the presenter controls.

Slide TypeBehaviorSuppressed Interactions
"web" Renders the URL as a static visual frame. Page is loaded, JavaScript executes, animations play, but the user cannot interact with the page. Scrolling, link navigation, form fields, text selection, pinch-zoom, context menus, video controls embedded in the page.
"pdf" Renders the PDF document at its initial view. The slide displays a single, stable frame for the slide's duration. Page turning, scrolling, pinch-zoom, text selection, internal link navigation, annotation interaction.
Implementation Note The Flutter runtime overlays an AbsorbPointer on top of the web view (ddl_web_player.dart) and the PDF viewer (ddl_pdf_player.dart) to consume all pointer events before they reach the embedded engine. On Linux, "web" slides open in an external window via desktop_webview_window because that backend does not support inline embedding; the in-show frame instead displays a placeholder.
Authoring Guidance If a target needs to be browsed interactively (multi-page navigation, scrolling content, or live forms), it does not belong in a DDL show. Use the host operating system's browser or PDF reader instead. For multi-page PDF presentations, export each page as an image slide, or split the PDF into separate single-page documents and chain them as consecutive slides.
DDLZ & Web Slides "web" slides require live network access and are not bundled into .ddlz archives (§12). A DDLZ archive transferred over the WebSocket protocol will play web slides only if the receiving display has internet connectivity to the referenced URL.

4.3 Remote Presenter Sync

A show in "presenter" mode can be used in a client / server setup over a local network. The mobile device acts as the remote (client) and a desktop or large screen acts as the display (server). The remote sends lightweight command messages via WebSocket; the display applies them.

Two operational models are supported:

  • Classic mode: Both devices load the same DDL show from their local asset bundles. The remote only sends navigation commands.
  • Server Demo mode: The display starts without a show and waits for the client to transfer the DDL (and optionally assets or a DDLZ archive). This allows a single mobile device to drive a display that has no pre-loaded content.

Five launch modes are available:

ModeRoleDescription
PlayStandaloneNormal local playback — no network required
DisplayServerStarts a WebSocket server on port 9130, shows the local IP, waits for a remote client, then plays the show controlled exclusively by the remote. All local user interaction is suppressed (displayOnly).
RemoteClientConnects to a display server by IP address. Runs the presenter mode locally and forwards all actions (slide navigation, laser pointer, timer toggle, video toggle) to the display via WebSocket.
Remote (Server Demo)ClientConnects to a display, starts a local HTTP asset server, rewrites asset:// URIs to http:// URLs pointing to the client, and sends the rewritten DDL via show.load. The display fetches assets over HTTP from the client.
Remote (Archive)ClientConnects to a display and sends a self-contained DDLZ archive via show.load_archive. The display extracts the archive to local storage, rewrites asset:// URIs to file:// paths, and plays the show. See §12.

5.3.1 Display-Only Mode

When the player runs as a display, the displayOnly flag is set. This suppresses all local user interaction:

  • Swipe navigation — disabled
  • Chevron buttons — hidden
  • Laser pointer (local) — disabled (remote laser is rendered)
  • Video double-tap pause/resume — disabled (remote controls playback)
  • Timer double-tap pause/resume — disabled
  • Timer widget — hidden by default (the timer is a presenter-only tool)
  • Controls overlay — suppressed

Only incoming RemoteCommand messages from the connected client can control the show.

5.3.2 Command Protocol

Commands are single-line JSON messages sent over WebSocket. Each has a type field and optional payload:

CommandPayloadDescription
slide.next—Advance to the next slide
slide.previous—Go back to the previous slide
slide.goto{ "index": 3 }Jump to a specific slide
show.load{ "ddl": "..." }Send a complete DDL JSON string to the display (server demo mode)
show.load_archive{ "archive": "...", "show": "name.ddl.json" }Send a base64-encoded DDLZ archive and the show file name to play. See §12.
laser.start{ "x": 0.45, "y": 0.32 }Begin laser pointer (normalized 0–1 coordinates)
laser.move{ "x": 0.47, "y": 0.33 }Update laser position
laser.end—Deactivate laser pointer
timer.toggle—Pause or resume the presenter timer
video.toggle—Pause or resume video playback
video.restart—Restart video from the beginning

Laser coordinates are normalized to 0.0–1.0 relative to the slide area, allowing cross-resolution sync between devices of different screen sizes. The laser position is clamped to the screen bounds on both sides.

5.3.3 Video Sound Routing

In remote client/server mode, the video_sound property controls which device plays the video audio track. This avoids double audio or echo when both devices are in the same room.

"video_sound": "server"     // "client" | "server" | "both"
ValueClient (Remote)Server (Display)Use Case
"client" (default)SoundMutedPresenter hears audio on their device; large screen is silent
"server"MutedSoundAudio plays through the display's speakers for the audience
"both"SoundSoundBoth devices play audio (e.g. when they are in separate rooms)

In standalone mode (no remote connection), video audio always plays regardless of this setting.

4.4 Motion Gesture Experimental

When mode is "presenter" and the app runs on iOS or Android, the optional motion_gesture feature enables slide navigation by physically flicking the device sideways. This allows hands-free slide advancement — for example when the presenter is walking around or holding a microphone.

Experimental This feature uses the device's hardware accelerometer and is labelled experimental. Behaviour may vary across devices and OS versions. The feature is opt-in and disabled by default.

The service uses the user accelerometer (gravity removed), so only motion-caused acceleration is measured. Only strong, deliberate movements that exceed the configured threshold trigger a slide change — normal phone handling, walking, or moderate gestures are ignored.

"motion_gesture": {
  "enabled":   true,    // Opt-in (default: false)
  "threshold": 15.0,    // Minimum acceleration in m/s²
  "cooldown":  600,     // Minimum ms between consecutive triggers
  "toggle": {           // Optional on-screen toggle button
    "enabled":  true,
    "position": "bottom-end",
    "offset_x": 0,
    "offset_y": -50
  }
}
PropertyTypeDefaultDescription
enabledbooleanfalseWhether motion gesture detection is active in presenter mode
thresholdnumber15.0 Minimum lateral acceleration in m/s² required to trigger a slide change. The user accelerometer (gravity removed) reports values relative to the device's own reference frame. Typical readings for reference:
  • 0–2 m/s² — phone sitting still or normal handling
  • 3–5 m/s² — walking, casual hand movements
  • 6–10 m/s² — moderate deliberate shake
  • 12–18 m/s² — harsh, forceful flick (recommended range)
  • 20+ m/s² — very violent movement
The default of 15.0 ensures only a strong, intentional sideways flick of the wrist triggers navigation. Lower values make the gesture more sensitive; higher values require more force.
cooldownnumber600Minimum interval in milliseconds between consecutive triggers. Prevents a single physical movement from firing multiple slide changes. Values below 300 are not recommended.
toggleobjectnullOptional on-screen toggle button to enable/disable motion detection at runtime. See sub-properties below.

Toggle Button

When toggle is provided, an on-screen button is rendered that lets the presenter enable or disable the motion gesture sensor at runtime. The button is positioned like a widget overlay and shows a visual indicator of the current state (active/inactive).

PropertyTypeDefaultDescription
toggle.enabledbooleanfalseWhether to display the toggle button
toggle.positionstring"bottom-end"Button position (same values as widget/caption positions: "top", "bottom", "center", "top-start", "top-end", "bottom-start", "bottom-end")
toggle.offset_xnumber0Horizontal offset in logical pixels
toggle.offset_ynumber0Vertical offset in logical pixels

Direction Mapping

The gesture interprets the x-axis of the user accelerometer (lateral axis when the phone is held upright in portrait orientation):

Physical MovementAccelerometer XAction
Flick device to the leftNegative spike (< −threshold)Next slide
Flick device to the rightPositive spike (> +threshold)Previous slide

In remote mode, the gesture emits slide.next / slide.previous commands over WebSocket, keeping the display in sync. The gesture is suppressed in displayOnly mode and in "auto" playback mode.

5.2 Dissolve / Pixelate

The "dissolve" transition divides the viewport into a grid of square blocks and reveals them in a pseudo-random order. This creates a distinctive pixelated reveal reminiscent of classic presentation software. The block size is configurable — smaller values produce a finer, smoother dissolve while larger values create a more dramatic mosaic effect.

{
  "type":       "dissolve",
  "duration":   1200,        // ms — longer durations recommended
  "block_size": 20           // px. Default: 20. Range: 4–100.
}
Implementation Note The player generates a deterministic shuffled order of block indices (seeded random) and clips the incoming slide to the subset of blocks that should be visible at the current animation progress. At progress 0 no blocks are visible; at progress 1 all blocks are shown and the clip path covers the full viewport.

5.3 Glitch

The "glitch" transition simulates a short digital disturbance during the slide change. The incoming slide appears with horizontal jitter (random displacement of the image along the x-axis) and chromatic aberration (colour channel shifts in the RGB matrix). The distortion intensity follows a bell curve — it is zero at the start and end of the transition and peaks at the midpoint, creating a natural "interference burst" effect.

{
  "type":      "glitch",
  "duration":  800,          // ms. Default: 600.
  "intensity": 20            // px max horizontal displacement. Default: 20.
}
Implementation Note The horizontal offset is computed from a stepped pseudo-random generator (quantised to ~20 discrete steps over the animation) to produce a stuttery, digital feel rather than smooth motion. A ColorFilter.matrix shifts the red, green and blue channels proportionally to the current intensity, creating the chromatic aberration overlay.

5.4 Blur / Gaussian Blur

The "blur" transition fades in the new slide while simultaneously animating a Gaussian blur from maximum sigma down to zero. This produces the impression of the image "focusing" into view out of a soft, dreamy haze. The sigma parameter controls the initial blur strength — higher values create a more dramatic, out-of-focus start; lower values yield a subtler softening.

{
  "type":   "blur",
  "duration": 1000,         // ms. Default: 600.
  "sigma":  20             // Max blur radius. Default: 20. Range: 1–60.
}
Implementation Note The Flutter runtime wraps the incoming slide in an ImageFiltered widget with ImageFilter.blur(sigmaX, sigmaY). The sigma is interpolated as maxSigma × (1 − progress), so the image starts fully blurred and sharpens to zero blur at progress 1. An opacity animation runs in parallel for a smooth cross-fade.

5.5 Zoom Reveal (Zoom In / Zoom Out)

The "zoom_reveal" transition scales the incoming slide from a point to full size (zoom in) or from an oversized view down to normal (zoom out). Unlike the gentler "zoom" transition which cross-fades, "zoom_reveal" performs a pure scale with no opacity change, producing a more dramatic cinematic zoom effect. The viewport is clipped so the scaling slide never overflows.

// Zoom in — image expands from center point
{
  "type": "zoom_reveal",
  "mode": "in",             // "in" (default) | "out"
  "duration": 800
}

// Zoom out — image shrinks from enlarged to normal
{
  "type": "zoom_reveal",
  "mode": "out",
  "scale_from": 3.0,        // Starting scale for out mode. Default: 1.2.
  "duration": 800
}
Implementation Note The incoming slide is wrapped in ClipRect to prevent overflow and a Transform.scale whose factor is driven by the animation progress. In zoom-in mode the scale runs from 0 → 1; in zoom-out mode it runs from scale_from → 1.

5.6 Iris / Shape Wipe

The "iris" transition reveals the new slide through an expanding geometric shape — evoking the classic "iris out" closing shot from early cinema and cartoons (famously used at the end of Looney Tunes episodes). The shape starts as an infinitely small point at the centre of the viewport and grows until it covers the entire frame.

{
  "type":     "iris",
  "shape":    "circle",      // "circle" (default) | "diamond" | "rectangle"
  "duration": 1000
}
ShapeDescription
"circle"Expanding circle from centre — classic iris wipe
"diamond"Expanding 45° rotated square (rhombus) from centre
"rectangle"Expanding rectangle preserving the viewport aspect ratio
Implementation Note A ClipPath with a custom clipper generates the shape path at each animation frame. The maximum radius equals half the viewport diagonal so the shape fully covers all four corners at progress 1.

5.7 Fade to Black / Fade to White

The "fade_color" transition fades the outgoing slide into a solid colour and then fades the incoming slide in from that same colour. The animation is split into two halves: during the first half (progress 0–0.5) the solid colour overlay rises to full opacity, hiding the old slide; during the second half (0.5–1) the new slide fades in over the solid colour. The color parameter accepts any hex colour value — "#000000" for a classic fade-to-black, "#FFFFFF" for fade-to-white, or any custom colour.

// Fade through black
{
  "type":     "fade_color",
  "color":    "#000000",     // Default. Any hex colour.
  "duration": 1200            // Longer durations recommended for this effect.
}

// Fade through white
{
  "type":     "fade_color",
  "color":    "#FFFFFF",
  "duration": 1200
}
Implementation Note The incoming child is wrapped in a Stack containing a ColoredBox and the slide content. The colour overlay opacity is (progress × 2).clamp(0, 1) — reaching full opacity at the midpoint. The slide content opacity is ((progress − 0.5) × 2).clamp(0, 1) — invisible until the midpoint, then fading in during the second half.

5.8 Curve Values

Any name from Flutter's Curves class is valid: "linear", "easeIn", "easeOut", "easeInOut", "bounceOut", "elasticIn", "decelerate", etc.

§6Ken Burns (Pan & Zoom)

The ken_burns1 object defines a slow animated pan and/or zoom applied to a still image while it is displayed. This technique gives static photographs a cinematic, living quality by smoothly moving across the image and gradually zooming in or out. It uses two anchor points — from and to — each specifying a focal point and zoom level as relative viewport fractions. The player animates from one state to the other over the full duration of the slide.

Background 1 The effect is named after American documentary filmmaker Ken Burns, who extensively used slow panning and zooming over still photographs in his acclaimed documentaries such as The Civil War (1990) and Baseball (1994). The technique became so closely associated with his work that Apple named the effect after him when they included it in iMovie (2003). While the name is widely recognised in North America, it may be less familiar to European audiences — technically, it is simply a pan-and-zoom animation applied to a still image. 
"ken_burns": {
  "from": { "x": 0.0, "y": 0.0, "scale": 1.0 },
  "to":   { "x": 0.1, "y": 0.05, "scale": 1.3 },
  "curve": "linear"
}
PropertyTypeDescription
from.x, from.ynumber [0.0–1.0]Starting focal point as fraction of image dimensions
from.scalenumber ≥ 1.0Starting zoom level (1.0 = no zoom)
to.x, to.ynumber [0.0–1.0]Ending focal point
to.scalenumber ≥ 1.0Ending zoom level
curvestringEasing curve name. Default: "linear"
Implementation Note The Flutter runtime should animate the image's transform matrix from the from state to the to state over the full duration of the slide, using a TweenAnimationBuilder or AnimationController with the specified curve.

§8Audio

Audio can be declared globally in defaults.audio (a continuous background track) or per-slide. Per-slide audio plays alongside the slide and stops on transition.

"audio": {
  "src":    "asset://audio/ambient.mp3",
  "volume": 0.8,                       // 0.0–1.0
  "loop":   true,                       // global track only
  "fade_in":  500,                      // ms fade-in at show start
  "fade_out": 1000                      // ms fade-out at show end
}

§9Widget Overlays

Widget overlays extend the caption concept to arbitrary named Flutter widgets that can be positioned over slides. They can be declared as an array in defaults.widgets (applied to every slide) and/or in individual slide objects. Per-slide widgets are merged with default widgets and rendered in a Flutter Stack ordered by z_index.

"widgets": [
  {
    "name":       "clock",          // Widget name (mapped to a Flutter widget)
    "parameter": {                   // Widget-specific parameters
      "color":  "#FFFFFF",
      "format": "hh:mm a"
    },
    "target":    "both",           // "client" | "server" | "both" (default)
    "position":  "top-end",        // Same positions as captions (§7)
    "offset_x":  0,                // Horizontal pixel offset from position
    "offset_y":  0,                // Vertical pixel offset from position
    "align":     "end",
    "animate_in": "fade",          // Same animations as captions
    "z_index":   1                 // Stack ordering (higher = on top)
  }
]

Unknown widget names are rendered as an info badge showing the widget name and its parameters, allowing developers to identify unregistered widgets during development.

Extensibility The widget system is designed for extensibility. Application developers can register custom widget builders via DdlWidgetRegistry to map DDL widget names to any Flutter widget.

§12DDLZ Archive Format

A DDLZ file is a standard ZIP archive (file extension .ddlz) that bundles one or more DDL show files together with all referenced assets (images, audio, Lottie animations, and PDF documents). This makes shows fully self-contained and portable — they can be transferred between devices without requiring a shared asset server or pre-installed content. Note that "web" type slides are not bundled as they require live internet access.

12.1 Archive Structure

The archive mirrors the asset directory layout used in DDL source URIs. All asset:// references in the DDL files map directly to paths inside the archive:

my_show.ddlz (ZIP)
├── diashows/
│   ├── nature.ddl.json          // DDL show files
│   ├── widget_demo.ddl.json
│   └── transition.ddl.json
├── images/
│   ├── paris.jpg                // Referenced via asset://images/paris.jpg
│   ├── alps.jpg
│   └── fireworks.json           // Lottie animations live alongside images
├── video/
│   └── intro.mp4               // Referenced via asset://video/intro.mp4
├── pdf/
│   └── sample.pdf               // Referenced via asset://pdf/sample.pdf
└── audio/
    └── ambient.mp3              // Referenced via asset://audio/ambient.mp3
Convention The directory names (diashows/, images/, video/, pdf/, audio/) mirror the Flutter asset directory structure. A DDL file referencing asset://images/dog.webp expects the file at images/dog.webp inside the archive.

12.2 Transfer Protocol

DDLZ archives are transferred from client to display via the show.load_archive WebSocket command (§4.3.2). The archive bytes are base64-encoded and sent as a JSON string alongside the name of the show file to play:

{
  "type":    "show.load_archive",
  "archive": "UEsDBBQAAAAIAG...",   // Base64-encoded ZIP bytes
  "show":    "widget_demo.ddl.json"  // Show file name inside diashows/
}

12.3 Extraction & Path Rewriting

When the display server receives a DDLZ archive, it performs the following steps:

  1. Extract — the ZIP archive is decoded and extracted to a temporary directory on the local filesystem (Directory.systemTemp/diashow_ddlz/).
  2. Locate show — the requested show file is read from diashows/{showName} inside the extracted directory.
  3. Rewrite URIs — all asset:// references in the DDL JSON are replaced with file:/// paths pointing to the extracted files. For example:
    asset://images/dog.webp → file:///tmp/diashow_ddlz/images/dog.webp
  4. Parse & play — the rewritten DDL JSON is parsed and the show begins playback.
Advantage over Server Demo Unlike the HTTP asset server approach (where the client must remain connected and serving assets throughout playback), the DDLZ archive transfers all data upfront. Once extracted, the display plays from local files and the client connection is only needed for navigation commands. This is more reliable over unstable WiFi connections and works even if the client's battery dies mid-show.

12.5 Creating a DDLZ

A DDLZ archive can be created with any standard ZIP tool. The only requirement is that the directory structure matches the asset:// URI paths used in the DDL show files:

# From the asset directory root:
zip -r my_show.ddlz diashows/ images/ video/ audio/

# Rename extension (optional — .zip works too):
mv my_show.zip my_show.ddlz
Size Considerations DDLZ archives are base64-encoded for WebSocket transfer, which increases the payload size by ~33%. For shows with many large images, the HTTP asset server approach (Server Demo mode) may be more efficient as it streams assets on demand. DDLZ is best suited for small to medium shows (under ~50 MB uncompressed).
